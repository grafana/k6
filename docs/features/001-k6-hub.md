# Feature Brief: k6 Hub - Unified Module Catalog

## Problem

k6 modules ship in three kinds - `builtin`s, compiled `extension`s, and remote `jslib`s - each with its own import syntax and versioning, and because the import path is tied to the implementation kind, a package that evolves across kinds changes its path and breaks scripts. Extension names are unscoped and first-come-first-served, the kinds are scattered (with community libraries catalogued nowhere), and nothing tells you which k6 version a module supports - so names get squatted, the ecosystem is hard to discover, and k6 upgrades break silently.

## Significance

These are five distinct sources of friction, each with a cost:

1. **Cognitive overhead** slows adoption. New users spend time learning module architecture rather than testing, raising the learning curve and the support burden.
2. **Import instability** actively discourages improvement. Every time a package moves from `jslib` → `extension` → `builtin`, user code breaks - so maintainers hesitate to optimize, and users hesitate to upgrade.
3. **Ungoverned namespaces** erode trust. Users can't tell an official Grafana package from a squatted community name by looking at the import, which blocks vendor branding and enterprise adoption where provenance matters.
4. **Poor discoverability** hides the ecosystem. Community libraries have no central catalog at all, so proven solutions go unfound and effort gets duplicated.
5. **Opaque k6 compatibility** makes upgrades risky. Nothing in the import tells you which k6 version a module supports - an `extension` works on only one k6 major, and `jslib`s carry no k6-version signal at all - so a k6 upgrade breaks scripts and finding a compatible version is trial-and-error.

The common root cause is that the import path is tied to *how* a module is implemented rather than *what* it is.

## Cost of inaction

The three-kind split keeps compounding: every new `jslib`, `extension`, or `builtin` widens the surface users must learn, and each optimization risks breaking scripts, so maintainers increasingly leave modules unoptimized to avoid churn. Namespace squatting gets harder to unwind the longer `k6/x/` grows, and the community library ecosystem stays invisible - proven utilities keep getting reinvented instead of shared. The ecosystem stays stuck paying the migration tax while it fragments further.

## Desired state

A single unified catalog (`hub.k6.io`) addresses every module by a scoped `k6:` URI, decoupling the stable import path from the implementation kind. A k6 script would look like this:

```javascript
// Proposed syntax

// latest
import { expect } from 'k6:@grafana/testing';
// exact release candidate
import { Faker } from 'k6:@grafana/faker' with { version: '1.0.0-rc.2' };
// compatible version range
import { flow } from 'k6:@gitlab/gpt' with { version: '^2.0.0' };
```

The Hub resolves the path to a concrete implementation, so the import path remains stable. Scopes (`@grafana`, `@gitlab`, `@nvidia`, etc.) give vendors identity and clear ownership, and one catalog covers every kind.

For a `builtin`-kind module the module version is implicit: it's whatever the k6 binary you're running ships, not something installed separately. A `version` constraint in the import is still matched against that module version, so a range that includes the shipped version resolves to it, while a constraint the shipped module version can't satisfy fails at load rather than resolving silently to a mismatched module.

From the user's point of view:

- **One import pattern** replaces three, with consistent version constraints.
- **Import paths stay stable** as a package evolves; users take compatible updates with a version range, or pin an exact version when they need a specific implementation.
- **Scoped names** signal source and support tier at a glance and let vendors publish under their own brand.
- **One place to search** all modules, official and community.
- **Explicit k6 compatibility** - each module version declares the k6 version(s) it supports in one catalog (extensions and jslibs alike), so upgrades resolve to a compatible version instead of breaking.
- **Runs regardless of kind** - imports resolve and execute across builtin, extension, and jslib alike; extension-kind modules are provisioned into a binary on demand by k6's build service, which the Hub subsumes from today's extension catalog.
- **Reproducible when you need it** - the Hub catalog maps each module version to an immutable commit, verifiable by digest, so a re-publish can't alter a pinned or archived run; pin an exact version or archive a test to freeze resolution, while ranges otherwise take the latest compatible update.

## Out of scope

- **Removing the existing import syntaxes.** The `k6:` URI adds a unified path; the current `jslib`, `extension`, and `builtin` imports keep working. Deprecating them is a separate decision.
- **Bit-for-bit extension binary reproducibility.** Pinning source commits is covered; fully pinning a built extension binary (k6 version, toolchain, co-bundled extensions) is a separate concern.
- **Private catalogs.** This brief covers a single public catalog only.
- **Unifying the authoring experience.** This brief unifies distribution and identity (one catalog, one namespace, one import pattern) across kinds; it does not collapse the kinds into a single authoring model. A `jslib` is pure JS, while an `extension` uses compiled Go to add native capabilities pure JS alone can't provide, so the kind a contributor builds follows from what the code needs. Streamlining the contributor path (publish flow and promotion between kinds) is a separate follow-up.
